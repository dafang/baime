# Proposal:自拓——范围生成引擎的 instrument 化与杠杆化

**日期**:2026-06-24
**状态**:奠基性讨论(proposal,非可直接拆 DoD 的工件)
**数据来源**:git log(302 commits 快照)、backlog(180 tasks)、docs/research、docs/experiments、meta-cc 会话历史
**关联文档**:[自指性分析](../baime-self-reference-analysis.md)、[几何信息论解读](../baime-geometric-information-theory.md)、[人类 GCL 负担与自举经济学](../baime-human-gcl-and-self-bootstrapping.md)、[GCL 完整观测机制](../research/gcl-complete-observation-mechanism.md)、[情境感知提案](proposal-situational-awareness.md)

> 本文提议把项目里一台一直被当作外生的引擎——**范围生成**——纳入方法论本身。
> 它不是又一个收敛式自动化,而是一个目标函数相反的发散式引擎;
> 把现有方法直接复制过去会掐死项目增长,必须是**扩展**。
> 本文论断尽量挂靠在 git/backlog/docs 可观测的谱系上;涉及推断处明确标注。

## 0. 命题

项目里有两台引擎,但只有一台被当成"工作"对待、被 instrument、被度量。

- **执行引擎(闭范围)**:把带 DoD 的 task/Epic 自动跑完。loop-backlog + agent + 验证 + merge。**收敛**。已高度自动化、已度量(GCL、DoD pass、七层验收)。
- **生成引擎(拓范围)**:把项目当前状态 + 人的项目外语境,变成**新范围**——proposal、Epic、研究假设、新方法论原语。元讨论、注意张力、引入外部类比。**发散**。几乎完全未 instrument、未度量,被当成"insight 从外面来"的外生输入。

人类参与者持续用 insight 刺激和推动项目,这些讨论"超越项目当时的范围"。这台引擎是当前真正的瓶颈和驱动器,却是整个方法论结构性的盲区。**本文主张:它应被纳入软件开发机制,并用与项目已应用的方法相近(但目标函数相反)的方法提效。**

---

## 1. 两台引擎,目标函数相反

| | 执行引擎 | 生成引擎 |
|---|---|---|
| 输入 | 带 DoD 的 task | 项目状态 + 人的项目外语境 |
| 产出 | 合入的代码 | 新范围(proposal/Epic/原语) |
| 人的角色 | **门**(create/promote/confirm) | **发生器本身** |
| 动力学 | 收敛(趋向不动点) | 发散(职责就是不收敛) |
| **优化目标** | **最小化人的介入** | **最大化人的杠杆** |

关键在最后一行:**两台引擎的目标函数相反。** 执行侧要把 GCL 往下压、趋向零人触;生成侧要的不是"更少的人类生成",而是"每个 insight 开更多范围"的更高产生成。

这条区分是整篇 proposal 的安全基座(见 §8)。

## 2. 形式化:执行求解 `f(x)=x`,生成改变 `f`

[自指性分析] 已把 OCA 刻画为不动点求解器:`f(method) = method`,观察→固化→自动化→新观察,在求迭代映射的不动点。这是**执行引擎**的精确描述——给定算子 `f`,对操作数 `x` 求解 `f(x)=x`。

**生成引擎不是不动点求解器,它是改变 `f` 本身的东西。** 它操作的是**算子,不是操作数**。

这一步差异解释了为什么生成无法套 DoD:DoD 预设了一个固定的 `f` 去对照检查。你无法写一条 `exit 0` 当"`f` 被改进成了更好的 `f`"的 shell 命令——因为"更好的 `f`"的判据本身是同一台引擎生成的。这是方法论内部的一处真实的哥德尔式不完备:**系统无法从内部完全验证自己的范围拓展。**

## 3. 项目历史中的生成引擎实例

生成引擎不是抽象设想——它已经在高速运行,只是没被命名和 instrument。从 git/docs/backlog 能重建出清晰的"insight → doc → proposal → Epic → 执行 → 新 insight"谱系。

### 实例 A:B″ 统一架构 = DreamCoder 压缩

观察到 `loop-meta` 与 `loop-backlog` 共享结构 → 合并为 B″ 单 worker(TASK-118/125,16 个子任务)。[几何信息论解读] 事后把这次合并重新框定为 **DreamCoder 的 abstraction sleep / MDL 压缩**——"发现两段逻辑可被同一抽象覆盖,于是压缩"。一个内部观察(两 skill 同构)生成了一次架构重构;一个外部框架(DreamCoder)又把它重新解释成 library learning。**生成的两半在这一个实例里都出现了:内部模式 + 外部框架。**

### 实例 B:GCL 螺旋(最完整的谱系)

```
gcl-definition(把"可理解程度"提成问题)
  → proposal-situational-awareness / TASK-150(量化:均值 14.55,gate 5.0)
  → premise-ledger / TASK-151(把 H 项从事后重建变成 gate 时自报)
  → gcl-selfReport-analysis / TASK-152(首批数据)
  → gcl-complete-observation-mechanism(升级为长期机制)
  → TASK-176 Epic(三阶段增量)
```

这是生成引擎产出图的一条完整螺旋:每个节点的执行产出,喂回下一轮生成。而且关键的一跳是**外部注入**:approval-fatigue 文献、EU AI Act、automation bias 把 GCL "从研究兴趣提升到系统安全属性"(见 [GCL 完整观测机制] §1)。这一跳**不可能从内部语料推导**——它来自人在项目之外读到的东西。

### 实例 C:几何信息论框架(深而慢、抗 Epic 转化)

GIT 假说 + DreamCoder 类比是纯外部注入,它把整个 OCA 重新框定为 wake-sleep library learning,并给出一个可证伪推论:"下一次 LLM 能力跃迁,会自动让再上一层变得划算。**问题只是那一层是什么。**"

这个框架**可能永远不成为一个 Epic**,却重塑了项目对自身的理解。它是 §5 度量悖论和 §8 标准危险的活样本:**用"转 Epic 率"度量生成,会把这类深而慢的框架判为低产,而它恰恰是最高杠杆的。**

### 实例 D:同一循环的不同半径(dogfooding → defect → cross-project)

[自指性分析] 记录:TASK-128/129 来自"用 loop-backlog 跑任务时暴露 loop-backlog 自身缺陷"。本周这条循环又扩了一个半径——跨项目缺陷 intake(TASK-179/180):baime 在 archguard/meta-cc 被消费,缺陷流回 baime backlog。

这两者和本文是**同一个循环 `执行产出 → 观察 → 生成新范围 → 执行` 的不同档位**:
- defect-intake = 半径最小的、纠正性的范围生成("修这个")
- insight-generation = 半径最大的、扩张性的范围生成("这暗示新方向")

而且在每个半径上,**人都在当传输总线**。上一轮我们在缺陷这个窄半径上把人从总线里拿出来了;本文问的是宽半径上能不能也拿。

### 实例 E:自指分析本身

[自指性分析] 这篇文档本身就是生成引擎的产物——它给了项目 **自指/自知/自举** 这套概念词汇。一次生成行为,产出的是项目用来理解自己的语言。本文(提出第四个"自")又是对它的一次生成,这层递归在 §6 处理。

### 实例 F:2026-06-24 会话（首个带量化数据的实时样本）

这次会话是第一个可以用 meta-cc 定量重建的完整生成-执行循环。

**量化对比**：

| | 生成引擎 | 执行引擎 |
|---|---|---|
| 时长 | ~8 小时 | 5 分钟（TASK-176 epic decompose） |
| 人工操作 | ~35 轮显式对话 | 2 次（promote + /loop-backlog） |
| 产出 | 6 epic + 1 proposal + grounding-infrastructure.md | 8 个子任务，R1 guard pass |
| 人的角色 | 发生器 | 门（触发后退出） |

**外部注入点（跳变，机器无法内推）**：
- "观察者隔离不是绝对规则"——由人从 BAIME 实践历史观察注入，触发 grounding §1 重写和 2×2 矩阵

- "micro-fix 的问题是人类注意力消耗，不是粒度"——问题框架翻转，触发跨项目 intake 方向

**光滑段（机器可插值）**：观测机制清单整理、gap 分析、epic-to-backlog 分解——全在已有框架内推断。

**速度不对称新发现**：执行引擎消费速度已跟不上生成引擎产出速度——6 个 epic 产出后全部停在 Epic: Backlog 等 promote，人的决策注意力成为瓶颈，而非执行能力。§7 方案 2（主动浮现 agent）的优先级因此上升：把"哪个 epic 先 promote"从人的注意力中移走。

**最外层递归的活体印证**："再次检查最近若干小时的讨论，这是一个关于更 meta 和更大颗粒度开发驱动机制的真实例子"——这句话本身是 §6 描述的最外层递归：要求系统观察自身生成活动的念头，来自人，不可能来自机器。且这次观察行为又是一次新生成（触发本次文档更新）。

## 4. 可机械化分解,与外部注入的不可替代性

把生成活动拆开,诚实地标:

| 子功能 | 可机械化? | 证据 |
|--------|-----------|------|
| 在语料里浮现候选 insight(模式、张力、跨文档连接) | **部分可** | 实例 A 的"两 skill 同构"是内部可发现的 |
| 注入外部框架(GIT/ANT/法规文献) | **不可** | 实例 B/C 的关键跳跃都来自项目外 |
| 选择哪个 insight 值得追 | **可脚手架,不可替代** | accept/reject 可记录成选择函数训练信号 |
| insight → 可执行范围(crystallize) | **已部分机械化** | epic-to-backlog / feature-to-backlog |
| 度量生成产出 | **完全没做** | 本文 §5 |

**更深的反思:外部注入点在 git 历史里表现为不连续。** 把项目的概念轨迹画出来,GCL 被 approval-fatigue 重新框定、OCA 被 GIT 重新框定——这些是**跳变**,不是内部语料能平滑推出的。内部语料挖掘(机器可做)产生轨迹的**光滑段**;外部框架注入(只有人能做)产生**跳变**。

> **机器能插值,只有人能外推到真正新的框架。** 这是分工的精确刻画,不是励志口号——它由项目历史的不连续点直接佐证。

## 5. 度量悖论:为什么生成只能滞后地、谱系式地度量

发散引擎不能用收敛量(它不该收敛)。但更深的问题是:**生成在它发生的当下,原则上不可度量。**

一次范围拓展的价值,是它下游解锁的一切的价值;而那只能通过尚未发生的执行实现。所以你**永远无法在生成行为发生时度量它**——只能事后、通过它的后代来度量。

这意味着:
- 生成引擎在**当下时态是结构性不可验证**的——这正是它装不进 DoD / 七层验收的根本原因(那些全是当下时态的检查)。
- 度量必须是**滞后的、谱系式的**(insight → 后代),而非当下的门。
- **任何当下时态的代理都会被 Goodhart。** 量"insight 数"得 insight-spam;量"转 Epic 率"压制深而慢的框架(实例 C)。

> 度量难,**不是因为还没找到指标,而是因为生成的当下度量在结构上不可能**。可用的只有谱系式滞后量:转化漏斗(insight→持久原语)、杠杆(单 insight 解锁的执行范围)、多样性(是否在自我重复——发散引擎的失败模式)。这三者数据其实已在(docs 交叉引用、task lineage),只是没聚合。

## 6. 自拓:第四个"自",与最外层的不可机械化

[自指性分析] 的三词:**自指**(描述自身)、**自知**(观测自身)、**自举**(改进自身)。

本文指的是第四个。自举是朝**给定方向**的收敛式自我改进(把已知目标执行得更好);本文说的是**生成方向本身**的发散式自我定向。建议命名 **自拓**(self-direction / self-expansion):项目给自己生成前沿。

它在"瓶颈搬家"论里是下一站:瓶颈从写代码(自举吃掉)→ 判断门(自知度量)→ 最上游的**范围生成**。人当下的高密度残差工作,大部分就是这台引擎。GIT 框架留下的开放问题"问题只是那一层是什么",正是自拓要回答的——**自拓是项目用来寻找自己下一层抽象的能力。**

**但有一处递归不可消除。** 决定"我们应该 instrument 范围生成"这件事,本身就是一次范围生成(就是这次对话)。所以自拓无法完全内化自己的顶端——**总有一个最外层的生成行为,由人未经 instrument 地完成。**

这不是待修的缺陷,是人不可替代的结构位置。对照 [自指性分析] 的"人退到 gate 上是稳定条件":在**执行**循环上人在**边界**;在**生成**循环上人在**最外层递归**。两者不同——边界可以被逼近和压缩,最外层递归不能,因为机械化它需要系统尚不具备的框架。

> **诚实的结论:生成螺旋除最外一圈外,皆可 instrument 和杠杆化;最外一圈永远是人、永远未机械化——这是必然,不是不足。**

## 7. 方案

三件可建之物,对应"给生成补 Observe / 脚手架生成 / 度量生成"。

**方案 1:Insight ledger(给生成补 Observe)。**
元讨论已是一等公民(docs/research/ + 交叉引用),补结构化元数据:触发它的张力/观察、注入了哪个外部框架、提议什么范围、以及 spawn 时回填的反链(是否长成 Epic/skill/ADR)。把生成产出变成**可度量的图**。投入最小——最接近已有的 docs/research + task lineage。

**方案 2:主动浮现 agent(脚手架生成,把人移到选择门)。**
周期性扫语料 + 近期 session,把内部矛盾、停滞假设、未开采模式作为**候选**抛给人选/否。accept/reject 喂回选择函数。这是 GCL 那套迁移**应用到生成**:把人从"从零生成所有 insight"移到"判断机器浮现的候选 + 补机器够不到的外部注入"。注意——它放大内部挖掘那一半,**不碰外部注入那一半**(§4)。

**方案 3:谱系式产出度量(发散引擎怎么量)。**
转化漏斗、杠杆、多样性,全部**价值滞后、抗早优化**(§5)。显式拒绝任何当下时态代理。

**分阶段:**
```
阶段 1:Insight ledger —— 让生成产出可见、可回填谱系
阶段 2:谱系度量 —— 在 ledger 上算转化/杠杆/多样性(滞后量)
阶段 3:主动浮现 agent —— 机器担内部挖掘,人担外部注入+选择
阶段 4:与执行闭环统一 —— 把 defect-intake(窄半径)和 insight-intake(宽半径)
        建在同一个 "执行产出→观察→生成→执行" 基质上
```
阶段一切口推荐 **方案 1**:投入最小,立刻让生成产出从隐性变可见。

## 8. 标准危险:目标函数倒灌

整个项目的机器(DoD、validate-plugin、ROI gate、GCL 最小化)都为**收敛/负荷削减**优化。存在一股引力,想把这套套到一切上——包括最终套到范围生成。

**危险很微妙且已几乎发生:** 执行引擎越好,"顺手把决定做什么也自动掉"的诱惑越大;而把负荷最小化套到生成上,会**collapse 项目的增长**——因为有价值的范围拓展依赖人的项目外语境(§4),把人从生成里"优化掉"等于切断外部注入。

> **必须显式给生成引擎设栅栏,把它隔离在负荷最小化目标之外。** 生成侧的目标是杠杆,不是消除。混淆二者是本提案要防的首要失稳。

---

## 9. 与现有方法的关系:扩展,而非复制

可复用的基质:OCA 的 Observe、meta-cc 观测、proposal→Epic 管道、度量纪律。
必须改变的:收敛→发散、当下门→滞后谱系量、人从"边界门"→"最外层递归"、目标从"最小化负荷"→"最大化杠杆"。

**自拓不是又一台收敛引擎,是一台目标相反的发散引擎,跑在同一套基质上。** 把这点记牢,能防止把发散引擎按收敛引擎优化、从而掐死增长(§8)。
