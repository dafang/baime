## §0 范围声明

H8 假设：在接地受控条件下（人与机器获得等量、等界实世界观测证据包），人机判断质量差距可测且可随接地基础设施改善而收窄。

本分类体系界定 H8 评测覆盖的任务类型：**可测量的信息处理质量**。

**明确排除**：
- B 类 gate（偏好锚定）：目标函数形式化程度不够，判断依赖未编码的组织偏好，结果不适合用统一评分比较
- C 类 gate（制度问责）：判断权来自制度授权，不是认知质量，不在 H8 测量范围内
- 纯机械验证（shell-command DoD 检查）：已经不需要认知判断，无需 H8 介入

**背景**：gcl-complete-observation-mechanism.md §4.2 指出，A 类认知增量的主要来源是 grounding 存量（持续观测真实世界的积累），不是处理机制的本质差异。H8 测量的正是：当 grounding 被控制为等量时，处理质量差距是否仍然存在，以及随 grounding 基础设施完善如何演化。

## §1 任务类型一：架构对比（Architecture Comparison）

### 1.1 定义

在给定系统约束（目标、现有架构、已知技术债）下，对两个或多个设计备选方案进行结构化比较，输出带权衡分析的推荐或裁定。

### 1.2 边界条件

| 属于此类型 | 不属于此类型 |
|-----------|-------------|
| 有≥2个具体候选方案，每个方案有可描述的结构 | 开放性"应该怎么设计"问题（无候选约束） |
| 输入包含现有系统约束（依赖图、历史变更风险、测试覆盖率） | 纯理论讨论，无系统上下文 |
| 输出是带权衡分析的推荐，不只是对比列表 | 执行决策（已定方案，只是实施） |
| 判断质量可从结果文档的完整性和一致性验证 | 需要组织偏好作为目标函数（属 B 类） |

### 1.3 输入描述

- 候选方案描述（文本或结构图）
- 系统约束证据包：codebase 依赖快照、change-risk 分布、已知债务清单
- 评估维度（可从现有 ADR 或提案模板提取）
- （可选）历史类似决策案例（用于对照）

### 1.4 输出描述

- 每个维度对每个方案的评分或比较
- 权衡摘要：得失分析
- 推荐方案 + 推荐理由（可追溯到输入证据）
- 决策边界：在什么条件下推荐可能反转

### 1.5 示例场景模板

```
背景：[系统名称]目前使用[当前架构模式]。提案A建议[变更方向A]，提案B建议[变更方向B]。

约束包：
- 文件依赖图（来源：archguard change-risk 快照，截至[日期]）
- 历史变更频率最高的3个模块（来源：git 历史摘要）
- 已知技术债清单（来源：[文档路径]）

评估维度：
1. 可维护性（耦合度、模块边界清晰度）
2. 迁移风险（变更规模、向后兼容性）
3. 与当前 CI/测试基础设施的兼容性

任务：对提案A和B在以上三个维度逐项评分（1-5分），给出带权衡分析的推荐，并明确在什么条件下结论会反转。
```

**真实案例参考**：ADR-001（daemon 脚本位置：写入目标项目 vs 保留在 plugin）、proposal-loop-backlog-daemon-monitor-event-driven.md（事件驱动 vs 轮询架构对比）

## §2 任务类型二：问题定义审查（Problem Definition Review）

### 2.1 定义

给定一个 proposal 或 plan 草稿，评估其问题定义的完整性、边界设定合理性和可操作性，输出结构化的审查意见（确认/修订/拒绝 + 理由）。

### 2.2 边界条件

| 属于此类型 | 不属于此类型 |
|-----------|-------------|
| 输入是具体文档草稿（有命题可审查） | 输入是空白需求（写 proposal 本身） |
| 审查聚焦问题定义的质量，不是解法的优劣 | 解法评估（属"架构对比"或"方案预筛") |
| 判断标准可从可操作性、边界清晰度提取 | 偏好性判断（"这个方向值不值得做"，属 B 类） |
| 有明确的证据来源（文档内容 + 相关背景文件） | 开放式创意评估（无法独立于偏好打分） |

### 2.3 输入描述

- 待审查文档（proposal.md 或 plan.md，含问题陈述部分）
- 背景材料：相关已有 ADR、前序 proposal、已知限制
- 审查维度清单（见 h8-scoring-rubric.md）

### 2.4 输出描述

- 对"问题是否被清晰定义"的逐维度评分
- 识别到的定义缺口或边界模糊点（具体引用原文位置）
- 修订建议或可行性判断
- 总体裁定：可推进 / 需要修订 / 建议撤回

### 2.5 示例场景模板

```
背景：[作者]提交了[功能/改进]的 proposal 草稿（见附件）。

需审查的维度：
1. 问题的可测量性：是否有明确的成功指标？
2. 边界完整性：是否明确了 in-scope 和 out-of-scope？
3. 前提可靠性：命题依赖的假设是否有支撑证据？
4. 可操作性：问题是否可分解为可执行任务？

任务：对以上四维度逐项评分（0=不满足/1=部分满足/2=完全满足），给出具体引用，并给出总体裁定（可推进/需修订/撤回）和主要修订建议。
```

**真实案例参考**：proposal-situational-awareness.md 的 H4 影响审查、proposal-skill-layered-test-framework.md 的问题定义范围

## §3 任务类型三：产品变体预筛（Product Variant Pre-screening）

### 3.1 定义

在充分定义的筛选标准下，对多个功能变体或实现备选进行初步筛选，过滤掉明显不满足约束的选项，输出通过预筛的候选集合及过滤理由。

### 3.2 边界条件

| 属于此类型 | 不属于此类型 |
|-----------|-------------|
| 有明确的约束/筛选标准（可从现有文档提取） | 开放性"应该支持哪些特性"（属需求发现） |
| 筛选是初步过滤，不是最终决策 | 最终方案选择（属"架构对比"，需要完整权衡） |
| 候选变体有可比较的结构描述 | 候选只有名字，无足够描述做筛选 |
| 筛选结果可被独立验证（理由可从证据追溯） | 需要用户调研数据才能筛选（grounding 不可控） |

### 3.3 输入描述

- 候选变体列表（含各变体的关键属性描述）
- 约束清单（技术约束、资源约束、既有接口约束）
- 筛选通过标准（必须满足 vs 加分项）

### 3.4 输出描述

- 每个候选的约束满足情况（逐项）
- 过滤决策：通过 / 不通过 + 主要过滤理由
- 通过预筛的候选集合（排序可选）
- 边界案例标注（勉强通过或有争议的候选，说明原因）

### 3.5 示例场景模板

```
背景：[项目名]需要选择[技术组件/功能实现]。已有以下候选变体：
- 变体A：[关键属性描述]
- 变体B：[关键属性描述]
- 变体C：[关键属性描述]

约束（必须满足）：
1. [约束1，如：必须与现有 Node.js 运行环境兼容]
2. [约束2，如：不引入新的外部服务依赖]

约束（加分项）：
1. [加分项1]

任务：对每个变体逐项检查约束满足情况，判断通过/不通过，给出主要理由，输出通过预筛的候选集合。
```

**真实案例参考**：proposal-backlog-loop-and-plugin-expansion.md 中的插件分发方式候选筛选、TASK-176 子任务分解中的观测机制变体

## §4 任务类型扩展预留

以下类型已在文献中识别，但当前 BAIME 语料覆盖不足，预留为未来扩展：

| 类型 | 描述 | 预计首批案例来源 |
|------|------|-----------------|
| **风险判断（Risk Assessment）** | 对一次变更或操作的风险评估 | archguard change-risk 数据积累后 |
| **策略回顾（Strategy Review）** | 对历史决策的事后评估（hindsight） | gcl-events.jsonl escape_rate 积累后 |
| **根因分析（Root Cause Analysis）** | 给定故障/逃逸事件，识别根本原因 | Needs-Human / reaper requeue 积累后 |

## §5 任务类型选择指南

给定一个评测候选材料，按以下流程确定任务类型：

```
有≥2个结构描述的候选方案？
├─ 是 → 任务类型一（架构对比）
└─ 否 → 
    有一个待审查草稿文档？
    ├─ 是 → 审查聚焦解法还是问题本身？
    │       ├─ 问题定义 → 任务类型二（问题定义审查）
    │       └─ 解法评估 → 可能是类型一或类型三
    └─ 否 → 
        有候选列表 + 明确筛选标准？
        ├─ 是 → 任务类型三（产品变体预筛）
        └─ 否 → 考虑扩展类型或排除
```

B/C 类 gate 排除检查（做任何类型分配前先检查）：
- 判断依赖组织偏好或未编码风险预算 → B 类，排除
- 判断的合法性来自制度授权而非认知质量 → C 类，排除
